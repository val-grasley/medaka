issue: 784
what: An interface method's signature carries a FREE, RETURN-POSITION effect var —
what: `speak : a -> <e> String`, where `e` is determined only by which impl DISPATCH
what: picks, not by any argument. `Speak Dog`'s body performs `<Stdout>` (`putStr`).
what: `pureUse : Speak a => a -> String` is declared PURE (no `<e>` at all) and its
what: body calls `speak x`. `pureUse`'s scheme quantifies nothing effect-wise, so the
what: checker instantiates the interface method's free `e` as `<>` and accepts —
what: nothing lower-bounds `e` to the impl's actual latent effect, because that
what: effect is selected by the DICT at the call site, invisible at `pureUse`'s own
what: definition. `medaka check`/`check --json` accept the program (exit 0, empty
what: diagnostics list) even though `pureUse` is certified pure yet dispatches to an
what: impl that performs `<Stdout>` — running it prints "[call] woof", IO from a
what: function the type system says is pure. No error at check, build, or run.
what: This is the RETURN-POSITION / free-effect-var variant of the laundering hole;
what: distinct from #782 (no `<e>` declared at all on the interface method — this
what: issue is *why* #782 can't be fixed by just adding `<e>`, since the var alone
what: doesn't soundly capture the effect) and from #781/#777 (impl-BODY argument-type
what: mismatch, unrelated to effects).
