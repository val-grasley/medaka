# Format: one extended-regexp per line (matched by `grep -E` against a failing

# logged as KNOWN-GAP, not a new finding.

# The Stage-0/1 MVP generator deliberately stays inside constructs that BOTH
# backends agree on, so this list is nearly empty by design.  The one entry

# NOTE on operator dispatch: Gap G (==/!=/</>/<=/>= dispatching to user/derived

# --- KNOWN GAP: derived Eq/Ord/Debug on a named-field `data` variant ---------
# `data T = C { f : .. } deriving (..)` : the selfhost tree-walker's deriving
# generator (selfhost/desugar.mdk showArm/conPatVars) builds a POSITIONAL
# `PCon C [..]` pattern for the impl body, which fails to match the VRecord
# value at eval time -> non-exhaustive panic at selfhost/eval.mdk:1050.  The
# OCaml oracle handles it (renders `C "A" 30`).  Filed here as a NEW divergence
# discovered by this work (see the fuzzer report); related docs:
#   - selfhost/CONSTRUCT-COVERAGE.md:189  "Record deriving (Debug) | GAP"
#     (documents the native-emitter & parser side; this is the distinct
#      tree-walker eval-time panic on the *derived impl pattern shape*)
#   - PLAN.md OBS4 (record CONSTRUCTION missing-field check — adjacent, distinct)
# The generator does NOT emit named-field data variants, so this should never
# fire; it is the classifier of record-bearing corpus/hand inputs.  Match is on
# the SOURCE: a `data ... = ... { ... } ... deriving` (named-field variant that
# derives), which is the only shape that triggers the tree-walker panic.
^(data|record).*[{].*[}].*deriving
